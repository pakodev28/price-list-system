# Система работы с прайс-листами и сметами

Веб-приложение для работы с прайс-листами поставщиков, каталогом товаров и сметами:
импорт данных из Excel с настройкой колонок, фоновый парсинг с прогрессом и
**автоматическое сопоставление позиций сметы с каталогом через ИИ** с цветовой
индикацией уверенности.

## Возможности

- **Поставщики**: CRUD, поиск по названию/ИНН, валюта.
- **Прайс-листы**: загрузка `.xlsx/.xls`, превью, настройка маппинга колонок, фоновый парсинг с прогрессом.
- **Каталог товаров**: артикул, наименование, ед. изм., группа — центральный узел сопоставления.
- **Проекты и сметы**: загрузка сметы из Excel, маппинг колонок (наименование, артикул, ед. изм., количество, цена материалов, цена монтажа), парсинг.
- **Сопоставление**: таблица позиций сметы, автосопоставление с каталогом, уверенность (зелёный/красный + %), ручная привязка и пометка «без соответствия».

## Стек

Django 5 + DRF · PostgreSQL 16 · Celery + Redis · python-calamine (Excel) ·
rapidfuzz + model2vec (локальная семантика) + опциональный LLM (OpenAI-совместимый) ·
React + TypeScript + Vite (в Docker — статическая сборка, раздаётся nginx) · Docker Compose.

## Быстрый старт

```bash
cp .env.example .env
docker compose up --build
```

- Фронтенд: <http://localhost:5173> (статическая сборка SPA на nginx)
- API + Swagger: <http://localhost:8000/api/docs/>
- Django admin: <http://localhost:8000/admin/> (примитивный «визуал» для данных)

Первая сборка бэкенда занимает пару минут (ставятся зависимости и запекается
модель эмбеддингов); фронтенд собирается за секунды (`npm ci` из lock-файла + `vite build`).

Демо-данные и пользователь админки:

```bash
docker compose run --rm backend python manage.py seed_demo
docker compose run --rm backend python manage.py embed_catalog   # векторы для семантического поиска
docker compose run --rm backend python manage.py createsuperuser
```

После `seed_demo` + `embed_catalog` откройте демо-смету в разделе «Проекты» и нажмите **«ИИ-сопоставление»**.

## ИИ-сопоставление

Каждая позиция проходит пайплайн (большинство решается локально, без сети):

1. **Артикул** — точное совпадение нормализованного артикула → уверенность 1.0.
2. **Гибридный поиск кандидатов** — лексика (`rapidfuzz`) ∪ семантика (локальные
   эмбеддинги `model2vec`, без GPU и без сети) с предварительной маршрутизацией по
   группам каталога. Доменная нормализация раскрывает аббревиатуры
   (Мор.→Морской, ЖД→железнодорожная, …), сближая каталог и «грязные» имена.
3. **Порог-гейт** — уверенный кандидат принимается сразу; слишком слабый →
   «без соответствия»; только спорные передаются в LLM.
4. **LLM-rerank** (опционально) — выбирает лучшего из спорных; при выключенном или
   недоступном LLM берётся топ ретрива.

Качество измеряется на размеченном наборе:
`docker compose run --rm backend python manage.py eval_matching` (recall@k + accuracy).

### LLM — опционально, два способа

По умолчанию `LLM_ENABLED=False` → приложение **полностью работает офлайн** (артикул +
лексика + локальная семантика), ключ не нужен. Чтобы включить LLM-rerank:

- **Вариант 1 — ключ автора.** Поставьте `LLM_ENABLED=True` и впишите в `.env`
  `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL`, которые предоставит автор.
- **Вариант 2 — свой ключ и своя модель.** Те же переменные можно направить на
  **любой OpenAI-совместимый** эндпоинт (OpenAI, TimeWeb, локальный vLLM/Ollama и
  т.п.) — со своим ключом и своей моделью.

```env
LLM_ENABLED=True
LLM_BASE_URL=<openai-совместимый эндпоинт оканчивающийся на /v1>
LLM_API_KEY=<ваш ключ>
LLM_MODEL=<имя модели>
```

Порог зелёный/красный задаётся `MATCH_THRESHOLD` (по умолчанию 0.75).

## Разработка

В Docker фронтенд — это прод-артефакт (статика на nginx), без dev-сервера. Для
локальной разработки с hot-reload запускайте Vite напрямую (API берётся с `:8000`):

```bash
cd frontend && npm install && npm run dev
```

## Тесты

```bash
docker compose run --rm backend pytest
```

Покрыты: нормализация, парсинг прайса, пайплайн матчинга, автосопоставление сметы, API поставщиков.
